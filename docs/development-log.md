> 自分の成長記録として残す開発ドキュメント

## 開発ステップ

### Step 1：企画・設計
- ゲームコンセプトを決定（スキル集め × エンジニア成長テーマ）
- アイテム・敵・レベル・ランクの仕様を設計
- パステル系のビジュアルイメージを固める

### Step 2：実装（index.html 1 ファイル）
- Canvas API でゲーム画面を構築
- プレイヤー移動・当たり判定・アイテムスポーンを実装
- 敵キャラ 3 種・エフェクト（パーティクル・画面揺れ）を追加
- HUD・タイトル画面・ゲームオーバー画面を作成

### Step 3：Git / GitHub へ公開
- `git init` でリポジトリを初期化
- `git add` → `git commit` でローカルに記録
- GitHub でリポジトリ（Public）を作成
- SSH キー設定（個人 GitHub アカウント用ホスト分離）
- `git push` で GitHub へアップロード完了

### Step 4：ドキュメント整備（現在）
- README.md を作成
- docs/spec.md（仕様書）を作成
- docs/development-log.md（本ファイル）を作成

## GitHub で練習したこと

| コマンド | 学んだ内容 |
|----------|-----------|
| `git init` | ローカルリポジトリの作成 |
| `git add` | ステージング（記録対象の選択） |
| `git commit -m` | 変更をローカルに記録、メッセージの書き方 |
| `git remote add origin` | リモートリポジトリとの接続 |
| `git branch -M main` | ブランチ名の変更 |
| `git push -u origin main` | GitHubへのアップロード、upstream の設定 |
| `git branch` | 現在のブランチを確認 |

## 学んだこと

### JavaScript
- `requestAnimationFrame` でなめらかなアニメーションを作る方法
- ゲームの状態（TITLE / PLAYING / GAMEOVER）を変数 1 つで管理する設計
- `filter()` で毎フレーム配列を更新する効率的な書き方
- スプレッド構文（`...def`）でマスターデータをコピーしてインスタンスを作る

### Git
- コミットは「作業の区切り」に行うものだとわかった
- SSH の設定が必要な理由（会社 PC で個人 GitHub を使う場合）
- `-u` オプションで次回から `git push` だけで済むようになる仕組み

### 設計・ドキュメント
- 仕様書は「作ったあとでも書ける」し、書くことで設計を言語化できる
- README はゲームの「顔」で、見た人が 30 秒で理解できることが大事
- マスターデータ（定義）とインスタンス（状態）を分けると変更しやすい
